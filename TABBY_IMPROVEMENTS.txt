WHY THE CURRENT DESIGN IS GOOD (keep these)
-------------------------------------------------------------------------------
- Drive-based emergent behaviour instead of a brittle state machine.
- Self-improving local/LLM split that trends API usage toward zero.
- Context-aware recall and reflection-into-facts are the right ideas.
- Clean three-file persistence separation.

THE FIVE WEAKNESSES THIS BRIEF FIXES
-------------------------------------------------------------------------------
W1 Recall similarity is crude (crc32 % 256 hashed bag-of-words). Heavy hash
   collisions, and similarity is dominated by shared template tokens, so
   "sleepy" lines and "hungry" lines look more alike than they are.
W2 Memory only grows in QUANTITY, never in QUALITY. Lines are never scored, so
   a line that always lands and a line that always falls flat are recalled
   equally. There is no eviction policy beyond a per-event cap.
W3 Coverage (and therefore the local-vs-API decision) is measured by line COUNT
   per event, not by how well the CURRENT context is covered. She can have 20
   greet lines and still have nothing good for "greet + scared + hungry".
W4 Behaviour never learns from the human. Affection rises, but behaviour
   selection does not adapt to which behaviours actually earn pets/feeds.
W5 No telemetry. The headline claim ("API use trends to zero") cannot be
   verified or tuned because nothing is measured.

PART A - MEMORY BRAIN (chatter.py)  -- make memory build up by QUALITY + COVERAGE
================================================================================

[M0] (P0) Add a simulation/replay harness FIRST so every change is verifiable.
  New file sim_harness.py:
    - Generate a synthetic stream of N events (default 1000) with plausible
      context distributions (vary mood/behaviour/drive buckets/affection).
    - Run them through the chatter with the LLM call STUBBED (returns a short
      canned line tagged by context) so no budget is spent.
    - Emit metrics: local_hit_rate over time, avg recall similarity of served
      lines, per-event coverage, distinct-lines-served ratio, repeat rate
      inside the anti-repeat window, mean reward of served lines.
  Acceptance: harness runs offline in < 5s and prints a before/after table.
  Use it to prove every item below moves the metric in the right direction.

[M1] (P0) Replace hashed-cosine recall with STRUCTURED context similarity.
  Keep the existing vector path behind flag RECALL_MODE = "structured" (default)
  | "vector" (legacy fallback).
  Represent context as a typed dict, not a flat string:
     ctx = {event, mood, behavior, energy_b, hunger_b, fear_b,
            affection_tier, daypart, claude_streak}
     - *_b buckets: "lo"/"mid"/"hi" (ordinal).
     - affection_tier: "wary"/"warming"/"bonded" from affection 0-100.
     - daypart: "night"/"morning"/"day"/"evening" from local hour.
     - claude_streak: "calm"/"flaky"/"productive" from recent claude events.
  Similarity (weighted field match, normalized to 0..1):
     FIELD_WEIGHTS = {mood:0.30, behavior:0.20, fear_b:0.20, energy_b:0.15,
                      hunger_b:0.15, affection_tier:0.10, daypart:0.10,
                      claude_streak:0.10}
     match(categorical) = 1.0 if equal else 0.0
     match(ordinal lo/mid/hi) = 1.0 if equal, 0.5 if adjacent, 0.0 if opposite
     sim = sum(w*match) / sum(w)
  Event stays a HARD filter (lines are per-event), with optional cross-event
  fallback in [M7].
  Why: interpretable, debuggable, collision-free, zero deps, far more accurate.
  Acceptance (via M0): avg recall similarity of served lines rises vs legacy.

[M2] (P0) Score every line and recall by REWARD, not just similarity.
  Extend each stored line:  {line, ctx, ts, reward:0.5, uses:0, last_used:0}
  The mascot already throttles/emits events; have it report an OUTCOME for the
  line it just showed, within a short window (e.g. 12s):
     positive: pet / feed / consoled shortly after  -> reward EWMA toward 1.0
     negative: scare, or quit/ignore with no interaction -> reward toward 0.0
     EWMA: reward = (1-a)*reward + a*signal,  a=0.25
  Recall weight becomes:
     w = (sim + 0.05)^3 * (0.5 + reward) * recency_boost * anti_repeat_penalty
  Sample from the top half as today, but with this weight.
  Why: she learns which lines LAND, so memory improves in quality over time.
  Acceptance: mean reward of served lines trends up across a sim run.

[M3] (P0) Make coverage (and local_prob) CONTEXT-aware, not count-aware.
  Replace event-count coverage with neighborhood coverage:
     neighbors = count(lines for this event with sim(ctx, line.ctx) >= 0.60)
     ctx_coverage = min(neighbors, COVER_CAP) / COVER_CAP
     local_prob = clamp(0.10 + 0.55*ctx_coverage + 0.45*budget_pressure,
                        0.0, 0.95)
  Effect: if the current context is well covered she answers locally; if it is a
  gap she spends an API call to FILL THAT GAP. Memory builds to cover context
  space evenly instead of piling up near-duplicates of common contexts.
  Acceptance: per-context coverage variance falls across a sim run; local hit
  rate still climbs overall.

[M4] (P1) Intelligent eviction (quality + diversity), replacing FIFO/cap.
  When an event exceeds LINE_CAP, drop the WORST line by:
     keep_score = 0.5*reward + 0.3*diversity + 0.2*recency
       diversity = mean(1 - sim(line, other_lines_for_event))
       recency   = normalized recency of last_used
  Never let an event collapse to a single line; keep at least MIN_KEEP=6.
  Why: retains high-performing, distinct lines; sheds dead/duplicate weight.

[M5] (P1) Anti-repetition window.
  Keep a per-session ring buffer of the last K=8 served line ids; set
  anti_repeat_penalty = 0.15 for lines in the buffer (do not hard-exclude, so a
  tiny memory still works). Clear on restart.
  Why: stops her repeating the same line back-to-back.

[M6] (P1) Voice-coherent prompts.
  When calling the LLM, include in the system/context:
     - top 2-3 highest-reward lines for this event as few-shot "her voice",
     - current user_facts (see R-items), the current context, MAX_LINE limit,
       and the ASCII-only constraint restated.
  Why: new lines stay on-voice and on-format; fewer rejects by _clean_line.

[M7] (P2) Cross-event recall fallback.
  If event has < MIN_KEEP usable neighbors, allow borrowing from a small set of
  compatible events (e.g. musing<->wake, claude_success<->done) at weight *0.5.
  Why: lifts coverage early, cuts API calls further. Guard with flag.

REFLECTION / LONG-TERM FACTS
[R1] (P1) Upgrade user_facts from a flat list to confidence-weighted records:
     fact = {text, category, evidence:int, confidence:0..1, last_seen}
     categories: "tools" | "schedule" | "temperament" | "style"
     On re-observation: evidence += 1, confidence -> toward 1.0 (EWMA).
     Decay confidence slowly with age; drop below DROP=0.2.
     Merge near-duplicate facts using the existing _similar() (> 0.8).
     Cap per category (e.g. 3) instead of one global cap of 8.
  Prompt includes only top-confidence facts per category.
  Why: facts stop going stale, stop duplicating, and read as real understanding.
[R2] (P2) Reserve a tiny reflection sub-budget (e.g. max 10% of DAILY_BUDGET) so
  reflection can never starve chatter, and skip reflection under budget pressure.

PART B - BEHAVIOUR BRAIN (brain.py)  -- make behaviour build up from experience
================================================================================

[B1] (P1) Mood inertia (smooth, not jittery).
  Maintain continuous valence (pleasant<->unpleasant) and arousal (calm<->excited)
  as EWMA of the drives + recent events; derive the mood LABEL from the smoothed
  values. Keep the same label set fed to the voice.
  Why: removes mood flicker when a drive sits on a threshold.

[B2] (P1) Behaviour reinforcement (she learns what earns attention).
  Add per-behaviour affinity (EWMA, init 0.5), persisted in cat_state.json.
  When a behaviour is followed within a window by pet/feed/consoled, nudge that
  behaviour's affinity up (esp. seek/play/beg); when followed by scare or
  nothing, nudge down. Selection weight becomes:
     weight = base * _factor(energy) * (0.7 + 0.6*affinity)   # affinity in 0..1
  Bound affinity to [0.2, 1.0] so innate personality (traits) still dominates and
  she cannot collapse onto one behaviour.
  Why: behaviour adapts to THIS human, which is the missing learning loop.

[B3] (P2) Coupled drives (more lifelike homeostasis).
  Add light coupling on top of linear drift:
     zoomies/play: energy drains faster AND hunger rises faster afterward.
     high fear: suppress hunger/social EXPRESSION (still drift, but urgency
                thresholds raised) until fear decays.
     grooming/nap completion: small valence bump (contentment).
  Keep deltas small; verify drives never runaway via the sim harness.

[B4] (P2) Trust + fear sensitization.
  Add trust (0..1, persisted). Consoling raises trust; a fresh scare with no
  recent console lowers it slightly. Dampen spikes:
     effective_scare = scare_amount * (1 - 0.5*trust)
  Track jumpiness that rises with repeated claude tool_failures in a short span
  and amplifies scare; decays over time.
  Why: an error storm makes her progressively jumpy; a patient human earns calm.

[B5] (P2) Day/night rhythm tied to the USER, not the clock.
  Learn the human's active hours from interaction timestamps (rolling histogram
  in cat_state.json). Modulate energy baseline so naps cluster in the user's
  quiet hours and she is livelier when the user is usually around.
  Why: she syncs to your schedule instead of a fixed circadian curve.

[B6] (P1) Recent-behaviour decay instead of single-step down-weight.
  Replace the flat just-used *0.15 with a short decaying history (last 4
  behaviours, penalty fades over ~30s) so variety feels natural, not random.

PART C - PERSISTENCE & SAFETY
================================================================================

[P1] (P0) Schema versioning + migration.
  Add "schema_version" to cat_state.json and cat_brain.json. On load, run
  migrate(old->new): add new fields (reward, uses, affinity, trust, fact
  records) with safe defaults; never lose existing data. Keep the self-heal
  re-clean + de-dup pass on load.

[P2] (P0) Telemetry file cat_metrics.json (fast-churn, gitignored).
  Append per day: api_calls, reflection_calls, local_hits, local_hit_rate,
  avg_served_sim, per_event_coverage, distinct_served_ratio, repeat_rate,
  mean_served_reward. Write atomically, never on the UI thread's hot path.
  Why: this is how you prove and tune "API trends to zero". Without it you are
  guessing. The sim harness [M0] reads the same schema.

[P3] Keep all writes atomic and UTF-8; new fields must round-trip cleanly so an
  old file opened by the new code, saved, and reopened is identical + migrated.
